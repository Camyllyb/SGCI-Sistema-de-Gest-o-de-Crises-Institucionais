1. Visão geral do sistema

O SGCI (Sistema de Gestão de Crises Institucionais) é uma aplicação web full-stack para registrar, acompanhar e encerrar crises em uma instituição de ensino (campi, departamentos), documentar as ações tomadas durante cada crise e emitir relatórios finais, com trilha de auditoria de todas as alterações.

O sistema é dividido em dois projetos independentes que se comunicam por HTTP/JSON:

Backend — API REST

Quarkus (Java 21). Expõe endpoints REST, aplica regras de negócio, persiste em PostgreSQL e cuida da autenticação por JWT em cookie HttpOnly.

groupId: br.com.crisismanagement
artifactId: crisis-management-api

Frontend — SPA

Angular 20 (100% standalone, com signals). Renderiza as telas, valida sessão a cada navegação e consome a API. Servido em dev por ng serve na porta 4200.

Conceitos do domínio

ConceitoDescrição
CriseOcorrência central. Tem título, descrição, nível de criticidade, status (ciclo de vida) e vínculos a tipo, campus, cenário e departamento.
Ação de criseRegistro cronológico do que foi feito durante a crise (contenção, comunicação, monitoramento, resolução).
Relatório de criseDocumento final único por crise, permitido só após a crise estar resolvida/encerrada.
Dados de referênciaInstituições, Campi, Departamentos, Tipos de Crise e Cenários — cadastros de apoio usados nos formulários.
AuditoriaLog imutável de toda operação de modificação (quem, o quê, quando, resultado HTTP).
Usuário / PerfilContas com perfil ADMIN (acesso total) ou COMUM (acesso restrito ao próprio departamento).

2. Arquitetura e stack tecnológica

Padrão arquitetural

O backend segue uma arquitetura em camadas clássica, com nomes que o próprio código documenta como BO (Business Object) para os services e DAO para os repositories:

Controller (JAX-RS / REST) → recebe HTTP, valida DTO, aplica @RolesAllowed ↓ Service (BO — regra de negócio) → orquestra, aplica regras, transações ↓ Repository (DAO — Panache) → acesso a dados / queries ↓ Entity (Hibernate ORM Panache) → mapeamento objeto-relacional ↓ PostgreSQL (schema versionado por Flyway)

A comunicação com o exterior usa DTOs (records Java): *Request para entrada (com Bean Validation) e *Response para saída — as entidades JPA nunca são expostas diretamente.

4. Modelo de domínio (entidades e enums)

Todas as entidades estendem PanacheEntityBase com campos públicos (padrão Panache) e id @GeneratedValue(IDENTITY). Os relacionamentos são modelados por chaves estrangeiras como Long (ex.: tipoCriseId), e não por associações JPA @ManyToOne — uma escolha de design que mantém as entidades simples e o controle das joins no serviço.

5. Banco de dados e migrações Flyway

O schema é gerenciado 100% por Flyway (quarkus.flyway.migrate-at-start=true); o Hibernate roda em modo validate (não altera tabelas, só confere que batem com as entidades). As migrações rodam em ordem na inicialização.

MigraçãoO que cria / altera
V1Tabela users (id, name, email único, password, active, created_at).
V2Tabelas departamentos e tipos_crise. Semeia 1 departamento ("Coordenação de TI"/CTI) e 1 tipo ("Tecnologia").
V3Tabelas instituicoes, campi (FK → instituicoes) e cenarios_crise (FKs → tipos_crise, departamentos).
V4Tabela crises com FKs para tipos_crise, campi, cenarios_crise (nullable) e departamentos.
V5Tabela acoes_crise (FKs → crises, users).
V6Tabela relatorios_crise — crise_id com constraint UNIQUE (1 relatório por crise).
V7Adiciona colunas perfil (default 'COMUM') e departamento_id em users; promove admin@crisis.local a ADMIN.
V8Tabela auditoria_logs + índices por data_hora DESC e por entidade_afetada.
Seed de aplicação (DataSeeder): na inicialização cria dois usuários se não existirem — admin@crisis.local / Admin@123 (perfil ADMIN) e usuario@crisis.local / Comum@123 (perfil COMUM, vinculado ao 1º departamento). As senhas são gravadas com hash BCrypt.

8. Segurança, autenticação e auditoria

Autenticação por JWT em cookie HttpOnly

O token não trafega no header Authorization: o Quarkus é configurado para lê-lo de um cookie chamado jwt. Isso protege contra XSS (o JS do browser não acessa o token).

# application.properties
mp.jwt.verify.issuer=https://crisis-management/issuer
mp.jwt.verify.publickey.location=publicKey.pem     # verifica assinatura
smallrye.jwt.sign.key.location=privateKey.pem       # assina no login
mp.jwt.token.header=Cookie
mp.jwt.token.cookie=jwt

Fluxo de login

  1. AuthService.login() busca o usuário por e-mail e confere active + BcryptUtil.matches().
  2. Emite JWT (RS256) com subject=id, claims name/email, groups=perfil e expiração de 1 hora.
  3. AuthController devolve o token em cookie via AuthCookieFactory: HttpOnly, SameSite=LAX, path=/, maxAge=3600.
  4. O logout devolve o mesmo cookie com maxAge=0 (expira).
return Jwt.issuer(issuer)
    .subject(user.id.toString())
    .claim("name", user.name).claim("email", user.email)
    .groups(user.perfil.name())          // "ADMIN" ou "COMUM" → role
    .expiresIn(TOKEN_EXPIRATION_SECONDS) // 3600s
    .sign();

Autorização

  • @Authenticated — exige sessão válida (qualquer perfil).
  • @RolesAllowed("ADMIN") — restringe a administradores; a role vem do claim groups.
  • @RolesAllowed({"ADMIN","COMUM"}) — ambos os perfis (ex.: registrar ação).
  • Regra de negócio adicional no service: usuário COMUM só registra ação em crise do próprio departamento (senão 403 Forbidden).

Auditoria automática — AuditoriaLogFilter

Um ContainerResponseFilter (JAX-RS) intercepta toda resposta e, para métodos de modificação (POST/PUT/PATCH/DELETE) feitos por usuário autenticado, grava um log com quem (subject/nome do JWT), a ação (CRIAR/ATUALIZAR/EXCLUIR), o método, a entidade derivada do path, o recurso e o status HTTP da resposta. Requisições anônimas e chamadas a auth/auditoria são ignoradas. Falhas na auditoria nunca interrompem a requisição original.

O registro roda em transação separada (@Transactional(REQUIRES_NEW)) porque o filtro executa após a transação da requisição original já ter sido encerrada.

Tratamento de erros

InvalidCredentialsException + InvalidCredentialsExceptionMapper convertem falha de login em 401 com corpo JSON (LoginResponse). Nos services, erros de regra usam exceções JAX-RS padrão: NotFoundException (404), BadRequestException (400), ForbiddenException (403).

CORS

Habilitado para o dev server Angular: origem http://localhost:4200, métodos GET/POST/PUT/DELETE/OPTIONS e allow-credentials=true (necessário para o cookie cruzar as origens).

9. Regras de negócio principais

RegraOndeComportamento
Ciclo de vida da criseCriseServiceNasce ABERTA; avança ABERTA → EM_ANDAMENTO → RESOLVIDA → ENCERRADA. Crise ENCERRADA não muda mais de status.
Ação por departamentoCriseService.addAcaoPerfil COMUM só registra ação em crise do seu departamentoId (senão 403).
Relatório único e tardioRelatorioCriseServiceSó para crise RESOLVIDA/ENCERRADA; máx. 1 por crise (constraint UNIQUE + checagem).
E-mail únicoUsuarioServiceNão permite dois usuários com o mesmo e-mail (criação e edição).
Senha na ediçãoUsuarioServiceSenha vazia/nula mantém a atual; preenchida gera novo hash BCrypt.
Auto-proteçãoUsuarioService.deactivateUsuário não pode desativar a própria conta.
Soft-deleteServices de referênciaDELETE marca active=false, preservando histórico.

Guards (revalidam no backend)

  • authGuard: chama GET /auth/me; sucesso → libera, erro → redireciona a /login. A sessão é revalidada no servidor a cada navegação (não confia em flag local).
  • adminGuard: idem, mas exige perfil === 'ADMIN'; autenticado sem permissão vai para /dashboard.

Interceptor de credenciais

export const credentialsInterceptor: HttpInterceptorFn = (req, next) =>
  next(req.clone({ withCredentials: true }));  // envia o cookie jwt cross-origin

Diretiva de papel — *appRole

Diretiva estrutural que renderiza o elemento só se o perfil do usuário estiver na lista permitida (*appRole="['ADMIN']"). Usa input.required() + effect() e reage automaticamente a login/logout via o signal currentUser. Governa a visibilidade do menu (a proteção real de rota fica nos guards + backend).

12. Frontend — funcionalidades (features)

Crises (núcleo)

Serviço com todos os endpoints: listAll, findById, create, changeStatus, listAcoes, createAcao.

  • Lista — carrega todas as crises no signal crises, com links para detalhe/criação.
  • Formulário (nova) — a tela mais rica: carrega em paralelo os 4 selects (tipos, campi, cenários, departamentos) e envia CriseRequest. É restrita a ADMIN no menu.
  • Detalhe — mostra a crise e suas ações; implementa a máquina de estados (fluxo: ABERTA→EM_ANDAMENTO→RESOLVIDA→ENCERRADA), expõe proximoStatus/avancarStatus() e só mostra o formulário de nova ação quando o status é ABERTA ou EM_ANDAMENTO.

Dashboard

Página única (landing pós-login). Consome GET /api/dashboard/resumo → DashboardResumo { totalCrises, crisesAbertas, crisesEmAndamento, crisesResolvidas, crisesCriticas }, exibindo KPIs que linkam para as crises.

Relatórios

list + form. No formulário, o select de crises é filtrado para exibir apenas crises RESOLVIDA/ENCERRADA (crisesElegiveis), espelhando a regra do backend.

Auditoria (só ADMIN)

Lista com filtros por entidade, ação e usuário (dropdowns vêm de /facets), badges coloridos por ação (CRIAR/ATUALIZAR/EXCLUIR) e por faixa de status HTTP, com DatePipe para a data/hora.

Dados de referência

Departamentos, Tipos de Crise, Instituições, Campi e Cenários. Os serviços expõem CRUD completo (incl. update e deactivate), porém, no estado atual, os componentes de lista apenas chamam listAll() — funcionando como listagens somente-leitura (não há rotas /nova ou /:id para eles). Apenas crises e relatórios têm telas de formulário.

13. Fluxos de ponta a ponta

Login

Login (Angular) --POST /auth/login--> AuthController AuthService valida BCrypt + emite JWT <--Set-Cookie: jwt (HttpOnly)-- navega para /dashboard Toda navegação seguinte: authGuard --GET /auth/me--> valida sessão

Ciclo completo de uma crise

1. ADMIN cria crise POST /api/crises (status ABERTA) 2. Equipe registra ações POST /api/crises/{id}/acoes (COMUM: só seu depto.) 3. ADMIN avança status PATCH /api/crises/{id}/status ABERTA → EM_ANDAMENTO → RESOLVIDA → ENCERRADA 4. ADMIN emite relatório POST /api/relatorios (só RESOLVIDA/ENCERRADA) * Cada POST/PUT/PATCH/DELETE gera 1 linha em auditoria_logs automaticamente.
