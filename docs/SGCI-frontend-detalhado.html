1. Panorama e princípios de design

O frontend do SGCI é uma Single Page Application em Angular 20.3, 100% standalone (sem NgModule), com estado reativo baseado em signals e um design system próprio em SCSS. Ele apenas renderiza as telas e consome a API REST do backend Quarkus.

Seis características que definem o front

#CaracterísticaConsequência prática
1100% standalone — todo componente declara seus próprios imports.Não há AppModule; o bootstrap é direto no componente raiz.
2Signals para estado (signal, computed, effect).Reatividade fina sem BehaviorSubject; a UI reage a login/logout automaticamente.
3inject() em vez de injeção por construtor.Código mais enxuto nos componentes/serviços.
4Lazy loading em toda rota (loadComponent).Cada tela vira um chunk carregado sob demanda.
5Sessão por cookie (HttpOnly), não token no JS.O cliente só espelha o usuário; a sessão real vive no servidor.
6Sem biblioteca de UI (nada de Material/Bootstrap).Design system manual em styles.scss, ícones SVG inline e gráficos SVG/CSS próprios.

2. Bootstrap e configuração global

A aplicação sobe sem módulos: main.ts monta o componente raiz App com um objeto de configuração. O App contém um <router-outlet> e monta uma única vez o <app-toast-container> (pilha global de notificações — ver seção 14).

// main.ts
bootstrapApplication(App, appConfig).catch((err) => console.error(err));

// app.ts — componente raiz
@Component({
  selector: 'app-root',
  imports: [RouterOutlet, ToastContainerComponent],  // toasts globais
  templateUrl: './app.html',
})
export class App {}

Todos os provedores globais ficam em app.config.ts:

// app.config.ts
export const appConfig: ApplicationConfig = {
  providers: [
    provideBrowserGlobalErrorListeners(),
    provideZoneChangeDetection({ eventCoalescing: true }),
    provideRouter(routes),
    provideHttpClient(withInterceptors([credentialsInterceptor])),  // cookie em toda req.
  ],
};

Pontos importantes:

  • provideRouter(routes) — registra as rotas lazy (seção 9).
  • provideHttpClient(withInterceptors([...])) — o HttpClient passa por um interceptor funcional que injeta as credenciais (o cookie de sessão) em toda requisição.
  • provideZoneChangeDetection({ eventCoalescing:true }) — detecção de mudanças baseada em Zone, agrupando eventos para reduzir ciclos.
A base da API está centralizada em core/config/api.config.ts: export const API_BASE_URL = 'http://localhost:8080'; — todos os serviços derivam suas URLs daí.

O pacote core reúne tudo que é transversal: config da API, guards, interceptor, diretiva de papel, os serviços de autenticação/toast, utilitários e os modelos base.

Política de senha (core/utils/password-policy.ts)

Regra compartilhada entre a troca (primeiro acesso e "Meu Perfil") e a redefinição de senha:

export const PASSWORD_MIN_LENGTH = 8;
export function validatePasswordPolicy(password: string): string | null {
  if (password.length < PASSWORD_MIN_LENGTH) return `A senha deve ter ao menos ${PASSWORD_MIN_LENGTH} caracteres.`;
  if (!/[A-Za-z]/.test(password)) return 'A senha deve conter ao menos uma letra.';
  if (!/[0-9]/.test(password))   return 'A senha deve conter ao menos um número.';
  return null;   // aprovada
}
Mín. 8 caracteres + ao menos uma letra + um número. A validação retorna uma mensagem amigável (ou null se ok) e está alinhada à política do backend (Bean Validation nos DTOs de senha).

Os tipos do front espelham exatamente os DTOs do backend — por exemplo, Perfil reflete o enum PerfilUsuario, e cada feature tem seus próprios tipos (ex.: StatusCrise, NivelCriticidade, TipoAcao) idênticos aos enums Java.

Utilitário de erro HTTP (core/utils/http-error.ts)

extractApiError(err, fallback) traduz uma HttpErrorResponse do Quarkus em mensagem legível: cobre Bean Validation (violations[]), erros de negócio ({ message }), corpos vazios (mensagem por status 401/403/404/409/500) e falha de rede (status 0). É usado por praticamente toda tela ao tratar subscribe({ error }).

5. Autenticação no cliente — AuthService

O servidor guarda a sessão no cookie; o cliente apenas espelha o usuário atual em um signal. Todo o resto da aplicação lê esse signal de forma reativa.

@Injectable({ providedIn: 'root' })
export class AuthService {
  private readonly http = inject(HttpClient);
  private readonly baseUrl = `${API_BASE_URL}/auth`;

  readonly currentUser = signal<User | null>(null);
  readonly isAuthenticated = computed(() => this.currentUser() !== null);

  login(credentials)      { return this.http.post<LoginResponse>(`${this.baseUrl}/login`, credentials); }
  logout()                { return this.http.post<void>(`${this.baseUrl}/logout`, {})
                              .pipe(tap(() => this.currentUser.set(null))); }   // limpa o espelho
  fetchCurrentUser()      { return this.http.get<User>(`${this.baseUrl}/me`)
                              .pipe(tap((u) => this.currentUser.set(u))); }  // atualiza o espelho

  getProfile()            { return this.http.get<Profile>(`${this.baseUrl}/perfil`); }   // Meu Perfil
  changePassword(request) { return this.http.post<LoginResponse>(`${this.baseUrl}/change-password`, request)
                              .pipe(tap(() => {                    // zera mustChangePassword no signal
                                const u = this.currentUser();
                                if (u) this.currentUser.set({ ...u, mustChangePassword: false });
                              })); }
  forgotPassword(email)   { return this.http.post<void>(`${this.baseUrl}/forgot-password`, { email }); }
  resetPassword(request)  { return this.http.post<LoginResponse>(`${this.baseUrl}/reset-password`, request); }
}
Padrão-chave: currentUser é a única fonte de verdade da sessão no cliente. Os guards a atualizam via fetchCurrentUser(); a diretiva *appRole e o layout a leem; o logout() a zera; e changePassword() a atualiza no lugar, apagando mustChangePassword — o que dispensa o modal bloqueante assim que a senha é definida.

Login — o componente

export class LoginComponent {
  readonly email = signal('');      readonly password = signal('');
  readonly errorMessage = signal<string | null>(null);
  readonly loading = signal(false);

  onSubmit(): void {
    this.loading.set(true);
    this.authService.login({ email: this.email(), password: this.password() }).subscribe({
      next: (r) => r.authenticated
        ? this.router.navigate(['/dashboard'])   // login sempre vai ao dashboard
        : this.errorMessage.set(r.message),
      error: () => this.errorMessage.set('Credenciais inválidas'),
    });
  }
}
Repare: o login não mais bifurca em "trocar senha". Mesmo com mustChangePassword, ele navega para /dashboard; é o layout autenticado que assume o controle e sobe o modal obrigatório (seção 15). A página de login ainda oferece o link "Esqueci minha senha" para /esqueci-senha.

6. Interceptor de credenciais

A API está em outra origem (8080) e a sessão é um cookie. Para o browser enviar esse cookie em requisições cross-origin, cada chamada precisa de withCredentials: true. Um interceptor funcional aplica isso a todas as requisições, de forma central:

export const credentialsInterceptor: HttpInterceptorFn = (req, next) => {
  return next(req.clone({ withCredentials: true }));
};

É o par natural da configuração de CORS do backend (access-control-allow-credentials=true): sem os dois lados alinhados, o cookie de sessão não trafegaria e toda requisição autenticada falharia com 401.

7. Guards de rota

Os guards são funções (CanActivateFn) e, em vez de confiar num flag local, revalidam a sessão no backend a cada navegação chamando GET /auth/me.

authGuard — exige sessão válida

export const authGuard: CanActivateFn = () => {
  const authService = inject(AuthService);
  const router = inject(Router);
  return authService.fetchCurrentUser().pipe(
    map(() => true),                                  // sessão ok → libera
    catchError(() => of(router.parseUrl('/login'))),   // erro → redireciona
  );
};
Mudança importante: o authGuard não redireciona mais quando mustChangePassword é verdadeiro. O primeiro acesso deixou de ser uma rota (/trocar-senha foi removida, junto do antigo changePasswordGuard): agora é um modal bloqueante montado no layout. O guard só valida a existência de sessão.

adminGuard — exige perfil ADMIN

export const adminGuard: CanActivateFn = () => {
  ...
  return authService.fetchCurrentUser().pipe(
    map((user) => user.perfil === 'ADMIN' ? true : router.parseUrl('/acesso-negado')),
    catchError(() => of(router.parseUrl('/login'))),
  );
};

Três desfechos: autenticado e ADMIN → segue; autenticado sem permissão → /acesso-negado (antes voltava a /dashboard); não autenticado → login. Como efeito colateral útil, fetchCurrentUser() também popula o signal currentUser, deixando a UI pronta assim que a rota carrega.

8. Diretiva estrutural *appRole

Controla a visibilidade visual de elementos conforme o perfil, reagindo a login/logout sem código extra — porque observa o signal currentUser dentro de um effect().

@Directive({ selector: '[appRole]', standalone: true })
export class RoleDirective {
  readonly appRole = input.required<string[] | string>();
  private rendered = false;

  constructor() {
    effect(() => {
      const roles = this.appRole();
      const allowed = Array.isArray(roles) ? roles : [roles];
      const user = this.auth.currentUser();
      this.toggle(!!user && allowed.includes(user.perfil));  // mostra/esconde
    });
  }
  // toggle: cria ou limpa a embedded view via ViewContainerRef
}

Uso no template: *appRole="['ADMIN']". Importante: isso é apenas cosmético — esconde botões, itens de menu, a busca/notificações do header e o formulário de ação da crise. A proteção real vem dos guards (front) e das anotações @RolesAllowed (back).

9. Roteamento e lazy loading

Todas as telas usam loadComponent (carregamento sob demanda). Há três rotas públicas (login e recuperação de senha) e o restante são filhas do MainLayoutComponent, que aplica authGuard uma vez para todas — e cada área administrativa reforça com adminGuard.

10. Layout principal (MainLayoutComponent)

É a casca de todas as telas autenticadas. Foi redesenhado: uma sidebar agrupada e recolhível, um header com breadcrumbs, busca global, central de notificações e menu de perfil, além de hospedar o modal bloqueante de primeiro acesso. O <router-outlet> renderiza a página atual no centro.

Sidebar agrupada e recolhível

Modal bloqueante de primeiro acesso

No fim do template do layout, condicionado ao signal do usuário:

12. Feature Crises em profundidade

12.1 Formulário de nova crise

É a tela mais rica: carrega em paralelo as quatro fontes dos selects e monta o CriseRequest. O nivel começa em BAIXO e as FKs começam nulas.

export class CriseFormComponent implements OnInit {
  readonly tipos = signal<TipoCrise[]>([]);   readonly campi = signal<Campus[]>([]);
  readonly cenarios = signal<Cenario[]>([]); readonly departamentos = signal<Departamento[]>([]);
  readonly niveis: NivelCriticidade[] = ['BAIXO','MEDIO','ALTO','CRITICO'];

  model: CriseRequest = { titulo:'', descricao:'', nivel:'BAIXO',
    tipoCriseId:null, campusId:null, cenarioId:null, departamentoId:null };

  ngOnInit() {                                     // popula os 4 dropdowns
    this.tipoCriseService.listAll().subscribe((d) => this.tipos.set(d));
    this.campusService.listAll().subscribe((d) => this.campi.set(d));
    this.cenarioService.listAll().subscribe((d) => this.cenarios.set(d));
    this.departamentoService.listAll().subscribe((d) => this.departamentos.set(d));

  onSubmit() {
    this.criseService.create(this.model).subscribe({
      next: () => this.router.navigate(['/crises']),
      error: () => this.error.set('Não foi possível cadastrar a crise...'),
    });

}
Criar crise é permitido a todos os autenticados (inclusive COMUM). No backend, a crise passa a ter dono (created_by, extraído do JWT); o COMUM só enxerga/acessa as próprias, enquanto o ADMIN vê todas (ver seções 17 e 18).

12.2 Detalhe da crise — máquina de estados no cliente

A tela de detalhe implementa o ciclo de vida da crise usando um mapa fluxo e dois computed: um para o próximo status, outro para decidir se o formulário de nova ação aparece.

private readonly fluxo: Record<StatusCrise, StatusCrise | null> = {
  ABERTA: 'EM_ANDAMENTO', EM_ANDAMENTO: 'RESOLVIDA',
  RESOLVIDA: 'ENCERRADA', ENCERRADA: null,          // terminal
};
readonly proximoStatus = computed(() => {
  const atual = this.crise();
  return atual ? this.fluxo[atual.status] : null;
});
readonly formVisivel = computed(() => {         // só registra ação se aberta/andamento
  const s = this.crise()?.status;
  return s === 'ABERTA' || s === 'EM_ANDAMENTO';
});

No template, tanto o bloco de avançar status quanto o formulário de nova ação ficam sob *appRole="['ADMIN']" — ou seja, o COMUM vê o detalhe e a timeline, mas somente leitura. Registrar ação (POST /api/crises/{id}/acoes) e mudar status são exclusivos do ADMIN.

Essa máquina de estados é o espelho da regra do backend (crise ENCERRADA é terminal); o front esconde os controles e o back garante de fato — defesa em profundidade. O sucesso das ações usa o ToastService.

Dashboard — um switch por perfil

Relatórios só ADMIN

14. Primitivos de UX (Toast, Modal, Skeleton)

Para padronizar feedback e evitar dependências externas, o núcleo ganhou um pequeno conjunto de primitivos reutilizáveis em core/components e core/services, apoiados por estilos globais em styles.scss (.toast-*, .modal-*, .skeleton, .field-error).

Toasts — ToastService (signals) + ToastContainerComponent

O serviço mantém a fila em um signal e oferece success(), error() e info(), cada um com auto-dismiss por setTimeout (4 s / 6 s). O container é montado uma única vez no App e renderiza a pilha no canto superior direito.

@Injectable({ providedIn: 'root' })
export class ToastService {
  private sequence = 0;
  readonly toasts = signal<Toast[]>([]);            // { id, type, message }
  success(m, t = 4000) { this.push('success', m, t); }
  error(m, t = 6000)   { this.push('error', m, t); }
  info(m, t = 4000)    { this.push('info', m, t); }
  dismiss(id)          { this.toasts.update((l) => l.filter((x) => x.id !== id)); }
  private push(type, message, ms) {
    const id = ++this.sequence;
    this.toasts.update((l) => [...l, { id, type, message }]);
    if (ms > 0) setTimeout(() => this.dismiss(id), ms);
  }
}

Modal genérico — ModalComponent

Overlay + cartão centralizado com projeção de conteúdo (<ng-content>). Inputs: title e closable (padrão true). Emite (close) ao clicar no botão de fechar ou no backdrop — mas quando closable=false ignora o backdrop e esconde o botão, viabilizando o fluxo obrigatório de primeiro acesso.

<div class="modal-overlay" (click)="onBackdrop()">
  <div class="modal-card" role="dialog" aria-modal="true" (click)="$event.stopPropagation()">
    <header class="modal-card__head">
      <h2>{{ title() }}</h2>
      @if (closable()) { <button (click)="close.emit()">✕</button> }
    </header>
    <div class="modal-card__body"><ng-content /></div>
  </div>
</div>
// onBackdrop(): só emite (close) se closable()

Skeleton e estilos de campo

Não é um componente, e sim uma classe utilitária global (.skeleton) usada para placeholders animados durante o carregamento, junto de .field-error/.field-hint para mensagens de formulário.

15. Primeiro acesso (modal bloqueante) e recuperação de senha

Primeiro acesso — modal obrigatório

Quando o backend marca mustChangePassword, o usuário faz login normalmente e cai no /dashboard; o layout então monta <app-change-password-modal [mandatory]="true">. Em modo obrigatório, o modal não é fechável (sem botão de cancelar, backdrop inerte) e exibe uma mensagem explicativa de primeiro acesso.

export class ChangePasswordModalComponent {
  readonly mandatory = input(false);
  readonly changed = output<void>();   readonly closed = output<void>();
  // título muda conforme o modo
  readonly title = computed(() => this.mandatory() ? 'Definição de senha obrigatória' : 'Alterar senha');

  onSubmit() {
    if (!this.currentPassword()) { this.errorMessage.set('Informe a senha atual.'); return; }
    const policyError = validatePasswordPolicy(this.newPassword());   // mín. 8 + letra + número
    if (policyError) { this.errorMessage.set(policyError); return; }
    if (this.newPassword() !== this.confirmPassword()) { this.errorMessage.set('A confirmação não corresponde…'); return; }
    this.authService.changePassword({ currentPassword, newPassword }).subscribe({
      next: () => { this.toast.success('Senha alterada com sucesso.'); this.changed.emit(); },
      error: (err) => this.errorMessage.set(extractApiError(err, '…Verifique a senha atual.')),
    });
  }
}

Como changePassword() zera mustChangePassword no signal, o @if do layout deixa de renderizar o modal automaticamente; em seguida (changed) dispara onFirstAccessChanged(), que navega para /dashboard. O mesmo componente, com mandatory=false, é reaproveitado em "Meu Perfil" (seção 16) com botão de cancelar.

Recuperação de senha — páginas públicas

Duas telas públicas (fora do layout autenticado) cobrem o fluxo de esquecimento:

  • /esqueci-senha (ForgotPasswordComponent): coleta o e-mail e chama forgotPassword(email). A resposta é sempre neutra — não revela se a conta existe: sucesso e falhas de servidor mostram a mesma tela de confirmação; apenas erro 400 (e-mail malformado) volta ao formulário com mensagem.
  • /redefinir-senha (ResetPasswordComponent): lê ?token= da query string no ngOnInit (sinaliza tokenMissing se ausente), valida a nova senha pela mesma política e chama resetPassword({ token, newPassword }). No sucesso, exibe confirmação e um toast, com link para voltar ao login; se o link expirou, mostra o erro via extractApiError.
No backend, o token é um hash SHA-256 guardado com validade de 30 min e o link chega por e-mail — do lado do front, tudo o que importa é ler o token da URL e respeitar a política de senha.

17. Centro de Monitoramento (admin) — AdminDashboardComponent

Para o ADMIN, o dashboard é um centro de monitoramento que agrega o estado do sistema em uma única tela densa. Toda a informação é computada no cliente a partir de endpoints ADMIN já existentes — nenhuma API nova foi criada.

19. Acesso Negado e endurecimento do perfil COMUM

Página /acesso-negado

Componente enxuto (template e estilos inline) exibido quando o adminGuard barra um usuário autenticado sem permissão. Mostra um ícone, a mensagem "Esta seção é restrita a administradores." e um link "Voltar ao início" para /dashboard.

Endurecimento do COMUM

O perfil COMUM foi restringido em três frentes, sempre com defesa em profundidade (front cosmético + backend autoritativo):

  • Detalhe da crise = somente leitura: o formulário de nova ação e o botão de avançar status ficam sob *appRole="['ADMIN']". O backend tornou POST /api/crises/{id}/acoes @RolesAllowed("ADMIN"); o COMUM ainda lê a timeline (GET …/acoes segue @Authenticated).
  • Sem Relatórios: o link some da sidebar (*appRole) e a rota /relatorios exige adminGuard; no backend o controller inteiro é @RolesAllowed("ADMIN").
  • Só as próprias crises: crises têm dono (created_by). O CriseService do backend filtra a listagem por dono para o COMUM e bloqueia o acesso ao detalhe de crise alheia com ForbiddenException; o ADMIN vê todas.

20. Fluxos de ponta a ponta

Autenticação e proteção de rota

1 Usuário envia login → AuthService.login() → POST /auth/login 2 Backend responde com Set-Cookie: jwt (HttpOnly) → navega SEMPRE para /dashboard 3 A cada navegação guardada: authGuard → fetchCurrentUser() → GET /auth/me (cookie enviado pelo interceptor) ├─ sucesso → popula currentUser + libera └─ erro → redireciona a /login 4 adminGuard (áreas ADMIN): exige perfil === 'ADMIN', senão vai a /acesso-negado 5 UI reativa: *appRole e o layout leem o signal currentUser 6 Logout → POST /auth/logout → currentUser.set(null) → /login

Primeiro acesso (modal bloqueante)

1 Login de usuário com mustChangePassword=true → vai para /dashboard normalmente 2 Layout detecta user().mustChangePassword → monta <app-change-password-modal [mandatory]="true"> (não fechável: sem Cancelar, backdrop inerte) 3 Usuário define nova senha (política: 8+ com letra e número) → changePassword() → POST /auth/change-password 4 Sucesso → toast + currentUser recebe mustChangePassword=false → @if some com o modal 5 (changed) dispara onFirstAccessChanged() → navega para /dashboard já liberado

Recuperação de senha (esqueci → e-mail → redefinir → login)

1 /esqueci-senha: informa e-mail → forgotPassword(email) → POST /auth/forgot-password (resposta NEUTRA — não revela se a conta existe; só 400 volta ao form) 2 Backend gera token (SHA-256, validade 30 min) e envia link por e-mail 3 Usuário abre o link → /redefinir-senha?token=... (ngOnInit lê o ?token=) 4 Define nova senha (mesma política) → resetPassword({ token, newPassword }) → POST /auth/reset-password 5 Sucesso → toast + tela de confirmação com link → /login

COMUM só enxerga as próprias crises

1 COMUM autentica → /dashboard mostra a home enxuta (KPIs das próprias crises) 2 /crises → criseService.listAll() → backend filtra por created_by do JWT 3 Abre /crises/:id própria → detalhe + timeline em SOMENTE LEITURA (form de ação e avançar status escondidos por *appRole="['ADMIN']") 4 Tenta crise alheia ou /relatorios → backend 403 / adminGuard → /acesso-negado 5 Pode criar crise (POST /api/crises grava created_by) — passa a ser dona dela

Operação sobre uma crise (ADMIN)

1 ADMIN em /crises/nova preenche o form (4 selects carregados em paralelo) → CriseService.create() → POST /api/crises → volta a /crises 2 Abre /crises/:id (detalhe + ações) 3 Registrar ação (form visível só se ABERTA/EM_ANDAMENTO e só ADMIN) → createAcao() → POST /api/crises/{id}/acoes → toast de sucesso 4 ADMIN avança status (computed proximoStatus) → changeStatus() → PATCH /api/crises/{id}/status 5 Após RESOLVIDA/ENCERRADA: /relatorios/novo lista só crises elegíveis
