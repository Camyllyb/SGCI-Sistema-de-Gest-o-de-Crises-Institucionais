1. Panorama e princípios de design

O backend do SGCI é uma API REST stateless construída em Quarkus 3.37.1 sobre Java 21. Ele concentra toda a lógica de negócio de gestão de crises e persiste os dados em PostgreSQL. O cliente Angular apenas consome os endpoints JSON.

Arquitetura em camadas

HTTP ─► Controller (JAX-RS) valida DTO (@Valid) e autorização (@RolesAllowed) Service (BO) regra de negócio, @Transactional Repository (DAO/Panache) queries e acesso a dados Entity (Hibernate ORM) mapeamento objeto ↔ tabela PostgreSQL schema versionado por Flyway

Os comentários do próprio código nomeiam o Service como BO (Business Object) e o Repository como DAO, deixando a separação de responsabilidades explícita.

Cinco decisões de design que definem o projeto

#DecisãoConsequência prática
1Panache Active Record: entidades com campos public, sem getters/setters.Código enxuto; acesso direto a crise.status.
2Sem associações JPA: relações são colunas Long (ex.: tipoCriseId), não @ManyToOne.Integridade referencial fica só no banco (FK do Flyway); nenhum cascade/lazy-loading.
3DTOs como record: *Request (entrada validada) e *Response (saída via from()).Entidades nunca vazam para a API; a senha nunca é serializada.
4Autenticação por JWT em cookie HttpOnly (não no header Authorization).Token inacessível ao JS → proteção contra XSS.
5Soft-delete: DELETE marca active=false.Histórico preservado; nada é apagado fisicamente.

2. Configuração da aplicação

Todo o comportamento de infraestrutura é declarado em src/main/resources/application.properties:

# App + servidor HTTP
quarkus.application.name=crisis-management-api
quarkus.http.port=8080

# Datasource PostgreSQL (porta 5433)
quarkus.datasource.db-kind=postgresql
quarkus.datasource.username=postgres
quarkus.datasource.password=postgres
quarkus.datasource.jdbc.url=jdbc:postgresql://localhost:5433/crisis_management

# Schema é do Flyway; Hibernate apenas valida (não altera tabelas)
quarkus.hibernate-orm.database.generation=validate
quarkus.flyway.migrate-at-start=true
quarkus.flyway.baseline-on-migrate=true

# JWT assinado (RS256) lido de COOKIE, não do header Authorization
mp.jwt.verify.issuer=https://crisis-management/issuer
mp.jwt.verify.publickey.location=publicKey.pem
smallrye.jwt.sign.key.location=privateKey.pem
mp.jwt.token.header=Cookie
mp.jwt.token.cookie=jwt

# CORS para o dev server Angular (credenciais on = cookie cruza origens)
quarkus.http.cors.enabled=true
quarkus.http.cors.origins=http://localhost:4200
quarkus.http.cors.access-control-allow-credentials=true
Combinação-chave: database.generation=validate + migrate-at-start=true significa que o Flyway é a única fonte de verdade do schema, e o Hibernate apenas confere, no start, que as entidades batem com as tabelas. Se divergirem, a aplicação nem sobe — um mecanismo de segurança.

3. Inicialização e semeadura — DataSeeder

Ao subir, o Quarkus dispara um StartupEvent. O DataSeeder o observa e cria, de forma idempotente (só se ainda não existirem), dois usuários iniciais com senha hasheada em BCrypt.

@ApplicationScoped
public class DataSeeder {
    @Transactional
    void onStart(@Observes StartupEvent event) {
        seedAdmin();
        seedComum();
    }
    private void seedAdmin() {
        if (userRepository.findByEmail("admin@crisis.local").isPresent()) return;
        User admin = new User();
        admin.name = "Administrador";
        admin.email = "admin@crisis.local";
        admin.password = BcryptUtil.bcryptHash("Admin@123");   // nunca em texto puro
        admin.perfil = PerfilUsuario.ADMIN;
        admin.active = true;
        admin.createdAt = LocalDateTime.now();
        userRepository.persist(admin);
    }
    // seedComum(): usuario@crisis.local / Comum@123, perfil COMUM,
    // vinculado ao 1º departamento existente (ou null).
}

4. Camada de entidades (Panache Active Record)

Todas estendem PanacheEntityBase, têm id @GeneratedValue(IDENTITY) e enums gravados como texto (@Enumerated(EnumType.STRING)). Repare que as FKs são Long puros:

@Entity @Table(name = "crises")
public class Crise extends PanacheEntityBase {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    public Long id;
    @Column(nullable = false, length = 200)  public String titulo;
    @Column(nullable = false, length = 1000) public String descricao;
    @Enumerated(EnumType.STRING) public StatusCrise status;
    @Enumerated(EnumType.STRING) public NivelCriticidade nivel;
    public Long tipoCriseId;   public Long campusId;       // FK por id
    public Long cenarioId;     public Long departamentoId;  // (não @ManyToOne)
    public boolean active;    public LocalDateTime createdAt;
}

7. Camada de serviços (BO) — regras de negócio

Todos @ApplicationScoped; métodos de escrita são @Transactional. Uma técnica recorrente: como as entidades são managed dentro da transação, alterar um campo já persiste — não é preciso chamar persist no update.

7.1 CriseService — o coração do domínio

Criar: a crise sempre nasce ABERTA e ativa, independentemente do que o cliente enviar:

@Transactional
public CriseResponse create(CriseRequest request) {
    Crise crise = new Crise();
    crise.titulo = request.titulo();  crise.descricao = request.descricao();
    crise.nivel = request.nivel();    crise.tipoCriseId = request.tipoCriseId();
    crise.campusId = request.campusId(); crise.cenarioId = request.cenarioId();
    crise.departamentoId = request.departamentoId();
    crise.status = StatusCrise.ABERTA;   // forçado pelo servidor
    crise.active = true;
    crise.createdAt = LocalDateTime.now();
    criseRepository.persist(crise);
    return CriseResponse.from(crise);
}

Mudar status: uma crise ENCERRADA é terminal — não muda mais:

@Transactional
public CriseResponse changeStatus(Long id, StatusCrise novoStatus) {
    Crise crise = getOrThrow(id);
    if (crise.status == StatusCrise.ENCERRADA)
        throw new BadRequestException("Não é possível alterar o status de uma crise encerrada.");
    crise.status = novoStatus;               // persistido pela transação
    return CriseResponse.from(crise);
}

Registrar ação: a regra de negócio mais importante — o usuário COMUM só atua no próprio departamento; o autor sai do JWT:

@Transactional
public AcaoCriseResponse addAcao(Long criseId, AcaoCriseRequest request) {
    Crise crise = getOrThrow(criseId);
    User autor = usuarioAtual();             // resolvido do subject do JWT
    if (autor.perfil == PerfilUsuario.COMUM
            && !crise.departamentoId.equals(autor.departamentoId))
        throw new ForbiddenException("Você só pode registrar ações em crises do seu departamento.");
    AcaoCrise acao = new AcaoCrise();
    acao.descricao = request.descricao(); acao.tipo = request.tipo();
    acao.criseId = crise.id; acao.usuarioId = autor.id;
    acao.createdAt = LocalDateTime.now();
    acaoCriseRepository.persist(acao);
    return AcaoCriseResponse.from(acao);
}
private User usuarioAtual() {
    Long id = Long.valueOf(jsonWebToken.getSubject());
    return userRepository.findByIdOptional(id).orElseThrow(...);
}

7.2 RelatorioCriseService — relatório único e tardio

@Transactional
public RelatorioCriseResponse create(RelatorioCriseRequest request) {
    Crise crise = criseRepository.findByIdOptional(request.criseId()).orElseThrow(...);
    if (crise.status != StatusCrise.RESOLVIDA && crise.status != StatusCrise.ENCERRADA)
        throw new BadRequestException("Só é possível gerar relatório de crises RESOLVIDAS ou ENCERRADAS.");
    if (relatorioCriseRepository.existsByCrise(crise.id))
        throw new BadRequestException("Esta crise já possui um relatório.");
    // ... preenche campos ...
    relatorio.usuarioId = Long.valueOf(jsonWebToken.getSubject());
    relatorioCriseRepository.persist(relatorio);
    return RelatorioCriseResponse.from(relatorio);
}

Dupla proteção contra duplicidade: checagem existsByCrise + constraint UNIQUE em crise_id no banco.

7.3 UsuarioService — CRUD administrativo com salvaguardas

  • E-mail único na criação e na edição (excluindo o próprio registro).
  • Senha: obrigatória ao criar; na edição, vazia/nula mantém a atual, preenchida gera novo hash BCrypt.
  • Auto-proteção: não é possível desativar o próprio usuário.
@Transactional
public void deactivate(Long id) {
    User user = getOrThrow(id);
    Long atual = Long.valueOf(jsonWebToken.getSubject());
    if (user.id.equals(atual))
        throw new BadRequestException("Você não pode desativar o próprio usuário.");
    user.active = false;                  // soft-delete
}

8. Camada de controllers (JAX-RS)

Fino por design: recebem HTTP, validam com @Valid, aplicam segurança e delegam ao service. A anotação de classe define a política padrão; métodos podem endurecê-la.

Padrão — CriseController

@Path("/api/crises") @Authenticated
@Consumes(APPLICATION_JSON) @Produces(APPLICATION_JSON)
public class CriseController {
    @Inject CriseService criseService;

    @GET public List<CriseResponse> listAll() { return criseService.listAll(); }

    @POST @RolesAllowed("ADMIN")
    public Response create(@Valid CriseRequest request) {
        return Response.status(CREATED).entity(criseService.create(request)).build();
    }
    @PATCH @Path("/{id}/status") @RolesAllowed("ADMIN")
    public CriseResponse changeStatus(@PathParam("id") Long id, @Valid StatusUpdateRequest r) { ... }

    @POST @Path("/{id}/acoes") @RolesAllowed({"ADMIN","COMUM"})
    public Response addAcao(...) { ... }
}

Política de acesso — resumo

Padrão do sistema: leituras liberadas a qualquer autenticado; escritas só ADMIN. Exceção: registrar ação de crise (ADMIN+COMUM). Usuários e auditoria são inteiramente ADMIN.

9. DTOs e validação (Bean Validation)

Entrada e saída são records. Os *Request declaram as regras de validação, disparadas pelo @Valid no controller; violações viram 400 automaticamente.

public record CriseRequest(
    @NotBlank(message="O título é obrigatório") @Size(max=200) String titulo,
    @NotBlank @Size(max=1000) String descricao,
    @NotNull  NivelCriticidade nivel,
    @NotNull  Long tipoCriseId,
    @NotNull  Long campusId,
              Long cenarioId,             // opcional (sem @NotNull)
    @NotNull  Long departamentoId) {}

Os *Response convertem a entidade via um método fábrica estático, garantindo que a entidade JPA nunca é serializada diretamente (e a senha jamais sai):

public record CriseResponse(Long id, String titulo, ..., LocalDateTime createdAt) {
    public static CriseResponse from(Crise c) {
        return new CriseResponse(c.id, c.titulo, ..., c.createdAt);
    }
}

UsuarioRequest ilustra validações compostas: @Email, @Size(min=6,max=100) na senha (opcional na edição) e @NotNull no perfil. O UsuarioResponse expõe id, name, email, perfil, departamentoId, active, createdAt — nunca a senha.

10. Segurança — JWT em cookie, passo a passo

O token não trafega em Authorization: Bearer; ele é lido de um cookie HttpOnly chamado jwt. Como o JavaScript não acessa cookies HttpOnly, o token fica protegido contra roubo por XSS.

Emissão do token — AuthService.login

public String login(String email, String password) {
    User user = userRepository.findByEmail(email).orElseThrow(InvalidCredentialsException::new);
    if (!user.active || !BcryptUtil.matches(password, user.password))
        throw new InvalidCredentialsException();
    return Jwt.issuer(issuer)
        .subject(user.id.toString())            // id → "quem sou"
        .claim("name", user.name)
        .claim("email", user.email)
        .groups(user.perfil.name())             // "ADMIN"/"COMUM" → role
        .expiresIn(TOKEN_EXPIRATION_SECONDS)     // 3600s = 1 hora
        .sign();                                 // assina com privateKey.pem (RS256)
}

Entrega no cookie — AuthController + AuthCookieFactory

// AuthController.login
return Response.ok(new LoginResponse(true, "Login realizado com sucesso"))
    .cookie(authCookieFactory.buildLoginCookie(token, TOKEN_EXPIRATION_SECONDS)).build();

// AuthCookieFactory — atributos de segurança do cookie
new NewCookie.Builder("jwt")
    .value(token).path("/")
    .httpOnly(true)                       // inacessível ao JS
    .sameSite(NewCookie.SameSite.LAX)         // mitiga CSRF
    .maxAge((int) maxAgeSeconds).build();

Fluxo completo

1 Cliente faz POST /auth/login (rota @PermitAll). 2 Servidor valida BCrypt e assina o JWT. 3 Devolve o token no cookie jwt HttpOnly. 4 Browser reenvia o cookie automaticamente a cada requisição. 5 SmallRye extrai/verifica o token do cookie; @Authenticated/@RolesAllowed autorizam. 6 POST /auth/logout devolve o cookie com maxAge=0, expirando-o.

11. Auditoria automática por filtro de resposta

Toda ação de modificação é registrada de forma transparente por um ContainerResponseFilter — os services não precisam chamar auditoria manualmente. Ele roda na resposta para capturar o status HTTP.

@Provider
public class AuditoriaLogFilter implements ContainerResponseFilter {
    @Override
    public void filter(ContainerRequestContext req, ContainerResponseContext resp) {
        if (!isModificacao(req.getMethod())) return;      // só POST/PUT/PATCH/DELETE
        String subject = jwt.getSubject();
        if (subject == null || subject.isBlank()) return;   // login anônimo não é auditado
        String entidade = extrairEntidade(normalize(req.getUriInfo().getPath()));
        if ("auth".equals(entidade) || "auditoria".equals(entidade)) return; // evita ruído
        try {
            auditoriaService.registrar(Long.valueOf(subject), nomeDoToken(subject),
                acaoDe(req.getMethod()), req.getMethod(), entidade,
                "/"+path, resp.getStatus());
        } catch (RuntimeException e) {
            LOG.warnf(e, "Falha ao registrar auditoria..."); // nunca quebra a requisição
        }
    }
    // POST→CRIAR, PUT/PATCH→ATUALIZAR, DELETE→EXCLUIR
}
Transação própria: o AuditoriaService.registrar é @Transactional(REQUIRES_NEW), porque o filtro executa depois que a transação da requisição original já foi encerrada. Assim o log é gravado numa transação independente e uma falha de auditoria não desfaz (nem depende de) a operação principal.

O que fica registrado em cada linha de auditoria_logs: quem (id + nome do token), ação amigável, método HTTP, entidade derivada do path, recurso completo, status HTTP e data/hora. Índices em data_hora DESC e entidade_afetada aceleram as consultas com filtro.

@Provider
public class InvalidCredentialsExceptionMapper
        implements ExceptionMapper<InvalidCredentialsException> {
    @Override public Response toResponse(InvalidCredentialsException ex) {
        return Response.status(UNAUTHORIZED).type(APPLICATION_JSON)
            .entity(new LoginResponse(false, ex.getMessage())).build();
    }
}

As demais exceções (NotFound, BadRequest, Forbidden) usam os mappers embutidos do Quarkus/RESTEasy — só o caso de credenciais tem mapper próprio, para devolver um corpo JSON amigável.

14. Ciclo de vida completo de uma crise (walkthrough)

Juntando todas as camadas, veja o caminho de ponta a ponta:

1 LOGIN POST /auth/login → AuthService valida BCrypt → JWT no cookie jwt 2 ABERTURA (ADMIN) POST /api/crises → @RolesAllowed(ADMIN) → CriseService.create() status = ABERTA (forçado) ; AuditoriaLogFilter grava CRIAR/crises/201 3 ATUAÇÃO (ADMIN ou COMUM) POST /api/crises/{id}/acoes → regra: COMUM só no próprio depto. nova AcaoCrise com autor = subject do JWT ; auditado 4 EVOLUÇÃO DO STATUS (ADMIN) PATCH /api/crises/{id}/status ABERTA → EM_ANDAMENTO → RESOLVIDA → ENCERRADA (ENCERRADA é terminal: bloqueia novas mudanças → 400) 5 RELATÓRIO FINAL (ADMIN) POST /api/relatorios → exige RESOLVIDA/ENCERRADA e 1 por crise 6 RASTREABILIDADE Cada POST/PUT/PATCH/DELETE gerou 1 linha em auditoria_logs consultável em /api/auditoria (só ADMIN)
