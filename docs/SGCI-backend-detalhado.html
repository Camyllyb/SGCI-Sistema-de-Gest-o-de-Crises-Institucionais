1. Panorama e princípios de design

O backend do SGCI é uma API REST stateless construída em Quarkus 3.37.1 sobre Java 21. Ele concentra toda a lógica de negócio de gestão de crises e persiste os dados em PostgreSQL. O cliente Angular apenas consome os endpoints JSON.

Arquitetura em camadas

HTTP ─► Controller (JAX-RS) valida DTO (@Valid) e autorização (@RolesAllowed) Service (BO) regra de negócio, @Transactional Repository (DAO/Panache) queries e acesso a dados Entity (Hibernate ORM) mapeamento objeto ↔ tabela PostgreSQL schema versionado por Flyway

Os comentários do próprio código nomeiam o Service como BO (Business Object) e o Repository como DAO, deixando a separação de responsabilidades explícita.

Cinco decisões de design que definem o projeto

#DecisãoConsequência prática
1Panache Active Record: entidades com campos public, sem getters/setters.Código enxuto; acesso direto a crise.status.
2Sem associações JPA: relações são colunas Long (ex.: tipoCriseId), não @ManyToOne.Integridade referencial fica só no banco (FK do Flyway); nenhum cascade/lazy-loading.
3DTOs como record: *Request (entrada validada) e *Response (saída via from()).Entidades nunca vazam para a API; a senha nunca é serializada.
4Autenticação por JWT em cookie HttpOnly (não no header Authorization).Token inacessível ao JS → proteção contra XSS.
5Soft-delete: DELETE marca active=false.Histórico preservado; nada é apagado fisicamente.

Combinação-chave: database.generation=validate + migrate-at-start=true significa que o Flyway é a única fonte de verdade do schema, e o Hibernate apenas confere, no start, que as entidades batem com as tabelas. Se divergirem, a aplicação nem sobe — um mecanismo de segurança.

3. Inicialização e semeadura — DataSeeder

Ao subir, o Quarkus dispara um StartupEvent. O DataSeeder o observa e cria, de forma idempotente (só se ainda não existirem), dois usuários iniciais com senha hasheada em BCrypt.

@ApplicationScoped
public class DataSeeder {
    @Transactional
    void onStart(@Observes StartupEvent event) {
        seedAdmin();
        seedComum();
    }
    private void seedAdmin() {
        if (userRepository.findByEmail("admin@crisis.local").isPresent()) return;
        User admin = new User();
        admin.name = "Administrador";
        admin.email = "admin@crisis.local";
        admin.password = BcryptUtil.bcryptHash("Admin@123");   // nunca em texto puro
        admin.perfil = PerfilUsuario.ADMIN;
        admin.active = true;
        admin.createdAt = LocalDateTime.now();
        userRepository.persist(admin);
    }
    // seedComum(): usuario@crisis.local / Comum@123, perfil COMUM,
    // vinculado ao 1º departamento existente (ou null).
}

4. Camada de entidades (Panache Active Record)

Todas estendem PanacheEntityBase, têm id @GeneratedValue(IDENTITY) e enums gravados como texto (@Enumerated(EnumType.STRING)). Repare que as FKs são Long puros:

@Entity @Table(name = "crises")
public class Crise extends PanacheEntityBase {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    public Long id;
    @Column(nullable = false, length = 200)  public String titulo;
    @Column(nullable = false, length = 1000) public String descricao;
    @Enumerated(EnumType.STRING) public StatusCrise status;
    @Enumerated(EnumType.STRING) public NivelCriticidade nivel;
    public Long tipoCriseId;   public Long campusId;       // FK por id
    public Long cenarioId;     public Long departamentoId;  // (não @ManyToOne)
    public Long createdBy;      // FK users: quem cadastrou (nulo pré-V10)
    public boolean active;    public LocalDateTime createdAt;
}

7. Camada de serviços (BO) — regras de negócio

Todos @ApplicationScoped; métodos de escrita são @Transactional. Uma técnica recorrente: como as entidades são managed dentro da transação, alterar um campo já persiste — não é preciso chamar persist no update.

7.1 CriseService — o coração do domínio

Criar: a crise sempre nasce ABERTA e ativa; além disso o servidor grava o dono (createdBy) a partir do usuário do JWT — é ele que habilita a "posse" das crises:

@Transactional
public CriseResponse create(CriseRequest request) {
    validarReferencias(request);
    User autor = usuarioAtual();             // resolvido do subject do JWT
    Crise crise = new Crise();
    crise.titulo = request.titulo();  crise.descricao = request.descricao();
    crise.nivel = request.nivel();    crise.tipoCriseId = request.tipoCriseId();
    crise.campusId = request.campusId(); crise.cenarioId = request.cenarioId();
    crise.departamentoId = request.departamentoId();
    crise.createdBy = autor.id;              // dono da crise (posse)
    crise.status = StatusCrise.ABERTA;   // forçado pelo servidor
    crise.active = true;
    crise.createdAt = LocalDateTime.now();
    criseRepository.persist(crise);
    return CriseResponse.from(crise, autor.name);
}

Listagem e detalhe são role-aware: o ADMIN enxerga todas as crises; o COMUM só as que ele mesmo cadastrou. O acesso a uma crise específica passa por um helper getAcessivel que lança ForbiddenException (403) se um COMUM tentar abrir a crise de outra pessoa:

public List<CriseResponse> listAll() {
    User atual = usuarioAtual();
    List<Crise> crises = atual.perfil == PerfilUsuario.COMUM
            ? criseRepository.listByCreator(atual.id)   // COMUM: só as próprias
            : criseRepository.listAllOrdered();         // ADMIN: todas
    Map<Long,String> nomes = nomesUsuarios(crises.stream().map(c -> c.createdBy).toList());
    return crises.stream().map(c -> CriseResponse.from(c, nomes.get(c.createdBy))).toList();
}
private Crise getAcessivel(Long id) {          // usado em findById / listAcoes / addAcao
    Crise crise = getOrThrow(id);
    User atual = usuarioAtual();
    if (atual.perfil == PerfilUsuario.COMUM && !atual.id.equals(crise.createdBy))
        throw new ForbiddenException("Você não tem acesso a esta crise.");   // 403
    return crise;
}

Resolução de nomes em lote: as respostas exibem nomes legíveis — createdByName na crise e usuarioNome em cada ação — resolvidos por uma única consulta (id in ?1) sobre o conjunto de ids:

private Map<Long,String> nomesUsuarios(Collection<Long> ids) {
    List<Long> distintos = ids.stream().filter(Objects::nonNull).distinct().toList();
    if (distintos.isEmpty()) return new HashMap<>();
    return userRepository.list("id in ?1", distintos).stream()
            .collect(Collectors.toMap(u -> u.id, u -> u.name, (a, b) -> a, HashMap::new));
}
Armadilha real (resolvida): o mapa de nomes usa HashMap de propósito — e nunca Map.of(). Crises anteriores à V10 têm createdBy nulo, e o resolvedor é consultado com nomes.get(null). Um Map.of() (mapa imutável do JDK) lança NullPointerException em get(null); o HashMap é tolerante a chave nula e apenas devolve null.

Registrar ação (hoje exclusivo do ADMIN — o endpoint POST /{id}/acoes passou a @RolesAllowed("ADMIN")): o autor sai do JWT e a crise é carregada por getAcessivel. A trava de departamento do COMUM permanece no BO como defesa em profundidade, mas na prática o COMUM nem alcança o método:

@Transactional
public AcaoCriseResponse addAcao(Long criseId, AcaoCriseRequest request) {
    Crise crise = getAcessivel(criseId);
    User autor = usuarioAtual();             // autor = subject do JWT
    // defesa em profundidade: COMUM só atuaria no próprio departamento
    if (autor.perfil == PerfilUsuario.COMUM && !crise.departamentoId.equals(autor.departamentoId))
        throw new ForbiddenException("Você só pode registrar ações em crises do seu departamento.");
    AcaoCrise acao = new AcaoCrise();
    acao.descricao = request.descricao(); acao.tipo = request.tipo();
    acao.criseId = crise.id; acao.usuarioId = autor.id;
    acao.createdAt = LocalDateTime.now();
    acaoCriseRepository.persist(acao);
    return AcaoCriseResponse.from(acao, autor.name);
}

Mudar status: uma crise ENCERRADA é terminal — não muda mais:

@Transactional
public CriseResponse changeStatus(Long id, StatusCrise novoStatus) {
    Crise crise = getOrThrow(id);
    if (crise.status == StatusCrise.ENCERRADA)
        throw new BadRequestException("Não é possível alterar o status de uma crise encerrada.");
    crise.status = novoStatus;               // persistido pela transação
    return CriseResponse.from(crise, nomeUsuario(crise.createdBy));
}

7.3 UsuarioService — CRUD administrativo com salvaguardas

  • E-mail único na criação e na edição (excluindo o próprio registro).
  • Senha: quando informada, tem no mínimo 8 caracteres (Bean Validation) e é hasheada em BCrypt, zerando mustChangePassword; na edição, vazia/nula mantém a senha atual.
  • Primeiro acesso: no create sem senha, o BO gera uma senha temporária (PasswordGenerator), marca mustChangePassword=true e envia por e-mail (EmailService.enviarSenhaTemporaria). O envio é resiliente: se o SMTP falhar, a conta fica criada e o admin pode reenviar via reset-senha.
  • Perfil estendido: create e update também gravam cargo e instituicaoId (além de departamentoId).
  • Auto-proteção: não é possível desativar o próprio usuário.
@Transactional
public void deactivate(Long id) {
    User user = getOrThrow(id);
    Long atual = Long.valueOf(jsonWebToken.getSubject());
    if (user.id.equals(atual))
        throw new BadRequestException("Você não pode desativar o próprio usuário.");
    user.active = false;                  // soft-delete
}

7.6 AuthService — perfil e recuperação de senha

Além de emitir o token no login (ver §10), o AuthService concentra a "Área do usuário": consulta de perfil, troca de senha autenticada e o fluxo público de recuperação por e-mail.

getProfile() — devolve um ProfileResponse com os nomes de departamento e instituição já resolvidos, além do cargo:

public ProfileResponse getProfile() {
    User user = usuarioAutenticado();
    String departamentoNome = user.departamentoId == null ? null
            : departamentoRepository.findByIdOptional(user.departamentoId).map(d -> d.nome).orElse(null);
    String instituicaoNome  = user.instituicaoId == null ? null
            : instituicaoRepository.findByIdOptional(user.instituicaoId).map(i -> i.nome).orElse(null);
    return new ProfileResponse(user.id, user.name, user.email, user.perfil, user.cargo,
            user.departamentoId, departamentoNome, user.instituicaoId, instituicaoNome);
}

changePassword() — valida a senha atual, exige que a nova seja diferente, zera a flag de troca obrigatória e reemite o token (o controller regrava o cookie jwt):

@Transactional
public String changePassword(String currentPassword, String newPassword) {
    // ... carrega o usuário do subject do JWT ...
    if (!BcryptUtil.matches(currentPassword, user.password))
        throw new BadRequestException("Senha atual incorreta.");
    if (BcryptUtil.matches(newPassword, user.password))
        throw new BadRequestException("A nova senha deve ser diferente da atual.");
    user.password = BcryptUtil.bcryptHash(newPassword);
    user.mustChangePassword = false;        // primeira troca zera a flag
    return issueToken(user);                  // reemite o cookie jwt
}

requestPasswordReset(email) — gera um token aleatório Base64url de 32 bytes, guarda apenas o SHA-256 do token (nunca o valor cru) com expiração de 30 minutos e dispara o e-mail com o link. A resposta ao cliente é sempre neutra — não revela se o e-mail existe (anti-enumeração de contas):

@Transactional
public void requestPasswordReset(String email) {
    userRepository.findByEmail(email).filter(u -> u.active).ifPresent(user -> {
        String token = gerarToken();          // 32 bytes aleatórios → Base64url
        user.resetToken = sha256(token);       // guarda só o HASH do token
        user.resetTokenExpiresAt = LocalDateTime.now().plusMinutes(30);
        emailService.enviarLinkRecuperacao(user.name, user.email, token, 30);
    });
    // sem else / sem exceção: o chamador recebe sempre a mesma resposta
}

resetPassword(token, novaSenha) — localiza o usuário pelo hash do token, valida a expiração, troca a senha (zerando também mustChangePassword) e limpa o token (uso único):

@Transactional
public void resetPassword(String token, String newPassword) {
    User user = userRepository.findByResetToken(sha256(token))
            .filter(u -> u.resetTokenExpiresAt != null
                    && u.resetTokenExpiresAt.isAfter(LocalDateTime.now()))
            .orElseThrow(() -> new BadRequestException(
                    "Link de recuperação inválido ou expirado. Solicite um novo."));
    user.password = BcryptUtil.bcryptHash(newPassword);
    user.mustChangePassword = false;
    user.resetToken = null;  user.resetTokenExpiresAt = null;   // uso único
}
Por que guardar só o hash? Se o banco vazar, o valor armazenado (SHA-256) não permite reconstruir o link válido — o token em claro só existiu no e-mail do usuário. Combinado à expiração curta e ao uso único, isso limita a janela de ataque.

8. Camada de controllers (JAX-RS)

Fino por design: recebem HTTP, validam com @Valid, aplicam segurança e delegam ao service. A anotação de classe define a política padrão; métodos podem endurecê-la.

Padrão — CriseController

@Path("/api/crises") @Authenticated
@Consumes(APPLICATION_JSON) @Produces(APPLICATION_JSON)
public class CriseController {
    @Inject CriseService criseService;

    @GET public List<CriseResponse> listAll() { return criseService.listAll(); }

    @POST @RolesAllowed({"ADMIN","COMUM"})   // COMUM também cria crises
    public Response create(@Valid CriseRequest request) {
        return Response.status(CREATED).entity(criseService.create(request)).build();
    }
    @PATCH @Path("/{id}/status") @RolesAllowed("ADMIN")
    public CriseResponse changeStatus(@PathParam("id") Long id, @Valid StatusUpdateRequest r) { ... }

    @POST @Path("/{id}/acoes") @RolesAllowed("ADMIN")   // registrar ação virou ADMIN-only
    public Response addAcao(...) { ... }
}

Política de acesso — resumo

Padrão do sistema: leituras liberadas a qualquer autenticado; escritas só ADMIN. Exceção: criar crise (ADMIN+COMUM). Crises e ações são filtradas por posse para o COMUM — ele só enxerga/acessa as próprias. Registrar ação e relatórios passaram a ser exclusivos do ADMIN. Usuários e auditoria seguem inteiramente ADMIN.

9. DTOs e validação (Bean Validation)

Entrada e saída são records. Os *Request declaram as regras de validação, disparadas pelo @Valid no controller; violações viram 400 automaticamente.

public record CriseRequest(
    @NotBlank(message="O título é obrigatório") @Size(max=200) String titulo,
    @NotBlank @Size(max=1000) String descricao,
    @NotNull  NivelCriticidade nivel,
    @NotNull  Long tipoCriseId,
    @NotNull  Long campusId,
              Long cenarioId,             // opcional (sem @NotNull)
    @NotNull  Long departamentoId) {}

Os *Response convertem a entidade via um método fábrica estático, garantindo que a entidade JPA nunca é serializada diretamente (e a senha jamais sai). O CriseResponse ganhou createdBy e createdByName; o AcaoCriseResponse ganhou usuarioNome — nomes resolvidos no BO:

public record CriseResponse(Long id, String titulo, ..., Long createdBy,
                           String createdByName, boolean active, LocalDateTime createdAt) {
    public static CriseResponse from(Crise c, String createdByName) {
        return new CriseResponse(c.id, c.titulo, ..., c.createdBy, createdByName, c.active, c.createdAt);
    }
}

UsuarioRequest ilustra validações compostas: @Email, @Size(min=8,max=100) na senha (opcional na edição), @NotNull no perfil e @Size(max=100) no cargo. O UsuarioResponse expõe id, name, email, perfil, departamentoId, instituicaoId, cargo, active, mustChangePassword, createdAt — nunca a senha.

Política de senha (backend): mínimo de 8 caracteres, validado por Bean Validation em ChangePasswordRequest, ResetPasswordRequest e UsuarioRequest (todos @Size(min=8, max=100)).

10. Segurança — JWT em cookie, passo a passo

O token não trafega em Authorization: Bearer; ele é lido de um cookie HttpOnly chamado jwt. Como o JavaScript não acessa cookies HttpOnly, o token fica protegido contra roubo por XSS.

Emissão do token — AuthService.login

public LoginResult login(String email, String password) {
    User user = userRepository.findByEmail(email).orElseThrow(InvalidCredentialsException::new);
    if (!user.active || !BcryptUtil.matches(password, user.password))
        throw new InvalidCredentialsException();
    return new LoginResult(issueToken(user), user.mustChangePassword);  // front decide se força troca
}
// issueToken(user):
Jwt.issuer(issuer)
   .subject(user.id.toString())            // id → "quem sou"
   .claim("name", user.name)
   .claim("email", user.email)
   .groups(user.perfil.name())             // "ADMIN"/"COMUM" → role
   .expiresIn(TOKEN_EXPIRATION_SECONDS)     // 3600s = 1 hora
   .sign();                                 // assina com privateKey.pem (RS256)

Entrega no cookie — AuthController + AuthCookieFactory

// AuthController.login
return Response.ok(new LoginResponse(true, "Login realizado com sucesso", result.mustChangePassword()))
    .cookie(authCookieFactory.buildLoginCookie(token, TOKEN_EXPIRATION_SECONDS)).build();

// AuthCookieFactory — atributos de segurança do cookie
new NewCookie.Builder("jwt")
    .value(token).path("/")
    .httpOnly(true)                       // inacessível ao JS
    .sameSite(NewCookie.SameSite.LAX)         // mitiga CSRF
    .maxAge((int) maxAgeSeconds).build();

Fluxo completo

1 Cliente faz POST /auth/login (rota @PermitAll). 2 Servidor valida BCrypt e assina o JWT. 3 Devolve o token no cookie jwt HttpOnly. 4 Browser reenvia o cookie automaticamente a cada requisição. 5 SmallRye extrai/verifica o token do cookie; @Authenticated/@RolesAllowed autorizam. 6 POST /auth/logout devolve o cookie com maxAge=0, expirando-o.

11. Auditoria automática por filtro de resposta

Toda ação de modificação é registrada de forma transparente por um ContainerResponseFilter — os services não precisam chamar auditoria manualmente. Ele roda na resposta para capturar o status HTTP.

@Provider
public class AuditoriaLogFilter implements ContainerResponseFilter {
    @Override
    public void filter(ContainerRequestContext req, ContainerResponseContext resp) {
        if (!isModificacao(req.getMethod())) return;      // só POST/PUT/PATCH/DELETE
        String subject = jwt.getSubject();
        if (subject == null || subject.isBlank()) return;   // login anônimo não é auditado
        String entidade = extrairEntidade(normalize(req.getUriInfo().getPath()));
        if ("auth".equals(entidade) || "auditoria".equals(entidade)) return; // evita ruído
        try {
            auditoriaService.registrar(Long.valueOf(subject), nomeDoToken(subject),
                acaoDe(req.getMethod()), req.getMethod(), entidade,
                "/"+path, resp.getStatus());
        } catch (RuntimeException e) {
            LOG.warnf(e, "Falha ao registrar auditoria..."); // nunca quebra a requisição
        }
    }
    // POST→CRIAR, PUT/PATCH→ATUALIZAR, DELETE→EXCLUIR
}
Transação própria: o AuditoriaService.registrar é @Transactional(REQUIRES_NEW), porque o filtro executa depois que a transação da requisição original já foi encerrada. Assim o log é gravado numa transação independente e uma falha de auditoria não desfaz (nem depende de) a operação principal.

O que fica registrado em cada linha de auditoria_logs: quem (id + nome do token), ação amigável, método HTTP, entidade derivada do path, recurso completo, status HTTP e data/hora. Índices em data_hora DESC e entidade_afetada aceleram as consultas com filtro. Rotas de auth (incluindo troca/recuperação de senha) são deliberadamente ignoradas para evitar ruído.

15. Área do usuário (perfil COMUM)

A "Área do usuário" reorganiza a experiência do perfil COMUM em torno de três pilares: onboarding por senha temporária, recuperação de senha autônoma e posse das crises que ele mesmo cadastra — com um endurecimento deliberado das permissões.

15.1 Primeiro acesso — senha temporária

Quando o admin cria um usuário sem informar senha, o UsuarioService gera uma senha temporária (PasswordGenerator), grava mustChangePassword=true e a envia por e-mail (EmailService.enviarSenhaTemporaria). No login, o backend devolve mustChangePassword no LoginResponse; o front trata isso como bloqueio até a primeira troca. A primeira chamada a POST /auth/change-password zera a flag e reemite o cookie jwt.

15.2 Recuperação de senha ("Esqueci minha senha")

Fluxo público em duas etapas, ambas @PermitAll:

  • POST /auth/forgot-password — recebe o e-mail e responde sempre 204 (resposta neutra, anti-enumeração). Se o e-mail existir e a conta estiver ativa, gera um token aleatório, guarda apenas o SHA-256 dele com expiração de 30 min e dispara o e-mail com o link /redefinir-senha?token=... (EmailService.enviarLinkRecuperacao).
  • POST /auth/reset-password — recebe token + newPassword, valida o hash e a expiração, troca a senha e limpa o token (uso único).

15.3 Posse de crises

A coluna crises.created_by (V10) marca o dono de cada crise. O ADMIN continua vendo e acessando todas; o COMUM só enxerga (listByCreator) e só acessa (getAcessivel) as que ele mesmo cadastrou — tentar abrir a crise de outra pessoa resulta em 403. Crises anteriores à V10 têm created_by nulo (daí o cuidado com HashMap na resolução de nomes, §7.1).

15.4 Endurecimento das permissões do COMUM

O COMUM deixou de registrar ações (POST /api/crises/{id}/acoes virou @RolesAllowed("ADMIN")) e não acessa relatórios (o RelatorioCriseController inteiro é @RolesAllowed("ADMIN"), leitura e criação). Ele continua podendo criar crises e ler a linha do tempo (GET /api/crises/{id}/acoes segue @Authenticated) das próprias crises.

15.5 Política de senha

Toda definição/troca de senha exige no mínimo 8 caracteres no backend (ChangePasswordRequest, ResetPasswordRequest, UsuarioRequest).
